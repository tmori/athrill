(3-2) 割込み処理に関する規定
	・割込みハンドラ番号と割込み番号の割当て，両者の対応
    　　プロセッサのベクタ番号をそのまま用いる。

	・割込み優先度の段階数（TMIN_INTPRIの値）
    　　デフォルトでは、TMIN_INTPRI=-15としている。（prc_kernel.h）
    　　変更する場合は、target_kernel.hでマクロTMIN_INTPRIを定義する。

	・dis_intとena_intのサポートの有無，その制限事項
    　　サポートしている。
    　　割込み禁止フラグがハードウェアでサポートされていないため、
    　　IPRxレジスタで割込み優先レベルを最低レベル（０）にすることに
    　　より割込み禁止フラグの機能を実現している。しかし、IPRxレジスタ
    　　では複数の割込み番号に対して、（概ね周辺モジュール単位に）一括
    　　して同一の割込み優先レベルを設定する仕様になっている。そのため、
    　　dis_int／ena_intを使用することにより、指定した割込み番号とは異
    　　なる割込み要求ラインについてもいっしょに割込み禁止／許可されて
    　　しまう。
    　　周辺モジュールについては、割込みコントローラではなく周辺モ
    　　ジュール側で割込み要因毎に割込み禁止／許可すれば、この問題を回
    　　避できるが、以下の理由でこの方法は採用していない。
    　　　・OSのオーバーヘッドが大きくなる。
    　　　・同じ割込み番号でも、周辺モジュールの設定によっては、割込み
    　　　　禁止／許可の方法が異なる。

	・CFG_INTの制限事項と拡張（ターゲット定義で使用できる割込み属性）
    　　IRQ割込みに関しては，以下の割込み属性を設定可能である．
    　　（内蔵周辺モジュール割込みに対しては設定できない。）
    　　　　・TA_NEGEDGE	ネガティブエッジトリガ
    　　　　・TA_POSEDGE	ポジティブエッジトリガ（SH2Aのみ）
    　　　　・TA_BOTHEDGE	両エッジトリガ（SH2Aのみ）
    　　　　・TA_LOWLEVEL	ローレベルトリガ
    　　
    　　IRQ割込みに対して、割込み属性を設定しない場合、デフォルトで
    　　ローレベルトリガとして設定される。
    　　
    　　IRQ割込みに対して、割込み属性として単にTA_EDGEを設定した場合
    　　（極性を設定しない場合）、ネガティブエッジトリガとして扱う。
    　　
    　　上記のdis_int／ena_intの制限事項と同じ理由で、下記のように割
    　　込み要求ラインの属性を設定した場合、割込み番号intno1とintno2
    　　がIPRxレジスタの同じ領域に割り当てられていると、(a)行の
    　　TA_ENAINT属性は反映されない。
    　　（カーネル動作開始時に割込み番号intno1の割込み要求ラインの割込
    　　　みが許可されていない。）

　　　　CFG_INT(intno1, { TA_ENAINT, intpri1 });  (a)
　　　　CFG_INT(intno2, { TA_NULL,   intpri2 });  (b)
　　　　ただし、intno1 < intno2 とする。

	・TOPPERS標準割込み処理モデルからのその他の逸脱
    　　・割込みハンドラ入口で必要なIRC操作 i_begin_int()
    　　　　エッジトリガの場合のみ、割込み要求をクリアするのが本則で
    　　　　あるが、トリガ種別（エッジトリガ／レベルトリガ）の判別を
    　　　　行うとオーバーヘッドが大きいので、どちらの場合もクリアし
    　　　　ている。
    　　　　現状では、SH2Aのみ実装されている固有の制限事項である。

(3-3) カーネル管理外の割込みに関する規定
	・TMIN_INTPRIは固定か設定できるか，その設定方法
    　　デフォルトでは、TMIN_INTPRI=-15としている。（prc_kernel.h）
    　　変更する場合は、target_kernel.hでマクロTMIN_INTPRIを定義する。

	・NMI以外にカーネル管理外の割込みを設けるか（設けられるようにするか）
    　　カーネル管理外の割込みをサポートする。

	・カーネル管理外の割込みの設定方法
	　　「TOPPERS新世代カーネル統合仕様書」の「2.7.8 カーネル管理外の
	　　割込みの設定方法」の「(b)カーネル管理外とする割込みをアプリ
	　　ケーションで設定できるようにする」を採用している。
	　　また、NMIについては、カーネル管理外の割込みの割込みに固定して
	　　いる。
	　　
	・カーネル管理外の割込みに対してDEF_INH，CFG_INTをサポートするか
    　　サポートする。

	・カーネル管理外の割込みハンドラが実行開始される時のシステム状態と
	　コンテキスト
	　　　・システム状態
	　　　　　・CPUロックフラグ
	　　　　　　　割込み受付時の値そのまま
	　　　　　・割込み優先度マスク
	　　　　　　　CFG_INTで設定した値
	　　　　　　　（NMIの場合は、-15に固定）
	　　　　　・ディスパッチ禁止フラグ
	　　　　　　　割込み受付時の値そのまま
	　　　・コンテキスト
	　　　　　非タスクコンテキスト
	　　　
	　　　また、スタックの切り替え処理も行う。（NMIを除く。）
	　　　
	・割込みハンドラの終了時に行われる処理
　　　　　割込み優先度マスク、コンテキストを元に戻す。
	　　　また、スタックの切り替え処理も行う。（NMIを除く。）

	・カーネル管理外の割込みハンドラの記述方法
	　　　カーネル管理の割込みハンドラと同様に通常のC言語関数として
	　　　記述できる。（引数、戻り値共にvoid型）

	・カーネル内の割込み出入口処理で起こる可能性のあるCPU例外
	　　CPUアドレスエラー
	　　
	・カーネル管理外の割込みに関する制限事項
	　　NMIの割込みハンドラが未登録の場合、デフォルトの割込みハンドラ
	　　が呼び出される。デフォルトの割込みハンドラは、トレースログ取
	　　得処理（log_inh_enterおよびlog_inh_leave）を呼び出しており、
	　　ポーティングガイド「6.6.7 カーネル管理外の割込み」にある
	　　「NMIの出入口処理では，トレースログ取得処理（log_inh_enterお
	　　よびlog_inh_leave）を呼び出してはならない」という規則から逸脱
	　　している。デフォルトの割込みハンドラからリターンすることはない
	　　（システム終了処理を呼び出す）ため、実質的な問題はないと考えて
	　　いる。

(3-7) その他
	・その他の制限事項
　　　　・FPUの扱い
　　　　　　本カーネルではSH2A-FPUのFPUはサポートしていない．

　　　　・コントロールレジスタの扱い
　　　　　　本カーネルではグローバルベースレジスタGBR、ジャンプテーブル
　　　　　　ベースレジスタTBRはサポートしていない．

　　　　・一般不当命令例外の扱い
　　　　　　一般不当命令例外の場合は戻り番地を2バイト進める必要がある
　　　　　　が対応していない。

　　　　・CPUロック状態のまま、割込みハンドラからリターンした場合、
　　　　　CPUロックを解除せずに、log_inh_leave()を呼び出してる。
　　　　　（割込み元に戻る直前にCPUロックを解除している。）
　　　　　同様にCPU例外ハンドラがCPUロックフラグを変更したままリターン
　　　　　した場合もCPUロックフラグを復元せずに、log_exc_leave()を呼び
　　　　　出してる。（オーバーヘッドが大きくなるため）

　　　　・IRQ割込み端子の初期化（SH2A固有）
　　　　　　本来はCFG_INTで指定されたIRQ入力端子の初期化をカーネル側で
　　　　　　すべきだが、SH2Aでは、IRQ番号から接続先の端子が一意に決ま
　　　　　　らない（複数の端子から選択できる）ため、IRQ入力端子の初期化
　　　　　　はサポートしていない。（端子番号に依存しない割込みコント
　　　　　　ローラの初期化は行っている。）

	・その他の拡張機能
    　　・未登録のCPU例外
    　　　　未登録の例外が発生した場合は，prc_config.c で定義されている
    　　　　default_exc_handler() が呼び出される．default_exc_handler()
    　　　　は，発生したCPU例外のベクタ番号をコンソールに出力した後，
    　　　　target_exit() を呼び出す．

    　　・未登録の割込み
    　　　　未登録の割込みが発生した場合は，prc_config.cで定義されている
    　　　　default_int_handler() が呼び出される．default_int_handler()
    　　　　は，発生した割込みの割込み番号をコンソールに出力した後，
    　　　　target_exit() を呼び出す．
    　　　　HRPカーネルの標準的なデフォルトの割込みハンドラ
    　　　　default_int_handler(void)は引数なし（ポーティングガイド
    　　　　porting.txt 「6.6.5 デフォルトの割込みハンドラ」参照）である
    　　　　が、SH12Aアーキテクチャ依存部では、カーネルから割込み番号を
    　　　　引数と渡されることを想定している。

    　　・シリアルドライバのHEWのI/Oシミュレーション対応
    　　　　「(9) その他の設定の変更方法」を参照

(9) その他の設定の変更方法

幾つかのパラメータは変更可能になっている．設定ファイル毎に設定可能項目
は次のようになっている．
Makefile.prcを編集した場合は、make realcleanとmake dependを再度、
実行すること。

●Makefileの設定項目
　　メモリマップに関する設定項目は「(5) メモリマップ」を参照。

    　　　　・マクロUSE_BANKED_REG（SH2A固有）
    　　　　　　マクロUSE_BANKED_REGが定義されると、割込みの出入口処理
    　　　　　　でレジスタバンクを使用する。
    　　　　　　定義場所：Makefile.sh2a　15行目付近
    　　　　　　　　　　　　CDEFS := $(CDEFS) -DUSE_BANKED_REG

    　　　　・マクロOMIT_POWER_CONTROL
    　　　　　　Makefile.targetで定義されることを想定している。
    　　　　　　ディスパッチャの出口で実行すべきタスクがない
    　　　　　　(schedtsk==NULL）場合は、sleep命令によってプロセッサを
    　　　　　　省電力モードに切り替えて割込み待ちをしている。
    　　　　　　マクロOMIT_POWER_CONTROLを定義すると、省電力モードに
    　　　　　　切り替えずにビジーループで割込みを待つ。
    　　　　　　（prc_support.Sの_dispatcher_2付近を参照）

    　　　　・マクロENABLE_CACHE
    　　　　・マクロCCR1
    　　　　　　（SH2A固有。ただし、sh7211を除く。）
    　　　　　　Makefile.targetで定義されることを想定している。
    　　　　　　ターゲット依存部でマクロENABLE_CACHEと定義すると
    　　　　　　start.S内にあるキャッシュの初期化ルーチンが有効になる。
    　　　　　　その際、ターゲット依存で、CCR1レジスタの初期値をマクロ
    　　　　　　CCR1_INITとして定義すること。
    　　　　
    　　　　　制限事項
    　　　　　　上記のキャッシュの初期化ルーチンは動作確認していない。
    　　　　　　（sh7211にはCCR1レジスタがないため）

    　　　　・マクロTOPPERS_HEW_SIMULATOR
    　　　　　　Makefile.targetで定義されることを想定している。
    　　　　　　このマクロが定義されると、シリアルドライバがHEWのI/O
    　　　　　　シミュレーションに対応したコードに置き換わる。
    　　　　　　シリアル送信はHEWのI/Oシミュレーション・ウィンドウに
    　　　　　　表示され（すべてポーリングになる）、シリアル受信はHEW
    　　　　　　のI/Oシミュレーションの擬似割込み機能により再現する。
    　　　　　　受信するキャラクタは、HEWのI/Oシミュレーション・ウィン
    　　　　　　ドウにカーソルを一番下の行に移動してから入力することに
    　　　　　　より指定する。（詳細はHEWのシミュレータ／デバッガのユー
    　　　　　　ザーズ・マニュアルを参照のこと）
    　　　　　　HEWのI/Oシミュレーション機能を利用するには、start.Sの
    　　　　　　ラベル_hew_io_sim_bodyをHEWに登録する必要がある。
    　　　　　　（ターゲットapsh2a_gccにおける動作確認時には0x0C00048E
    　　　　　　　番地を登録した。）
    　　　　　　
    　　　　　　設定箇所
    　　　　　　　基本設定メニュー
    　　　　　　　　　シミュレータ
    　　　　　　　　　　システム
    　　　　　　　　　　　「システム」タブ
    　　　　　　　　　　　　　I/Oシミュレーション・アドレス
    　　　　　　　　　　　　　（「有効」チェックボックスで有効を指定する）

    　　　　　　備考
    　　　　　　　HEWのシミュレータは、sleep命令の実行でブレークする
    　　　　　　　仕様になっている。（割込み待ちにならない。）
    　　　　　　　これでは、時間経過によるタイマ割込みがシミュレート
    　　　　　　　できないので、マクロTOPPERS_HEW_SIMULATORが定義され
    　　　　　　　る場合は、マクロOMIT_POWER_CONTROLも定義し、
    　　　　　　　prc_support.Sの_dispatcher_2付近のsleep命令を実行し
    　　　　　　　ないようにしている。

●target_config.hの設定項目

　　・割込みを許可するldc命令の遅延について
　　　　以下のマクロを定義する。
　　　　・LDC_NOP_DISPATCHER
    　　　　　実行すべきタスクが存在しない場合の割込み発生待ちに挿入
    　　　　　するnop命令
    　　　　　　・マクロOMIT_POWER_CONTROLが定義され、sleep命令を使用
　　　　　　　　　しない場合に有効である。
　　　　　　　　・キャッシュ無効時も必要に応じて定義する。
　　　　　　　　　（アクセス速度の速いSRAM上に命令コードを配置する場合
　　　　　　　　　　など）
　　　　　　　定義場所：target_config.h
　　　　・LDC_NOP_SET_SR
　　　　　　　set_sr()に挿入するnop命令
　　　　　　　定義場所：target_config.h
　　　　・LDC_NOP_SIL_SET_IIPM
　　　　　　　SIL_UNL_INT()に挿入するnop命令
　　　　　　　定義場所：target_sil.h

    　　詳しくはprc_design.txtの「●割込み許可区間が短い場合の注意事項
   　　（SH2Aコア固有）」およびターゲット依存部のtarget_user.txtを参照
